# Wallet MVP Design

## Scope

Wallet is a family/group-oriented envelope budgeting product. The first release is a PWA, deployed on Coolify, with a clean API boundary so a future React Native app can reuse the backend.

In scope for the MVP:

- Email/password accounts.
- Group creation and shareable invite links.
- Owner/admin/member roles.
- Persistent envelope budgeting with carry-forward balances.
- Manual expenses, manual envelope funding, and envelope-to-envelope transfers.
- Recurring expenses as upcoming expected expenses/reminders.
- Shared dashboard, envelope list, activity feed, reports, and settings.
- Installable PWA with offline read-only cached screens.
- Coolify deployment with PostgreSQL.
- Bun as the package manager, script runner, test runner, and default JavaScript runtime.

Out of scope for the MVP:

- Bank sync or automatic transaction import.
- Member-to-member debt settlement.
- Offline writes and conflict resolution.
- Native React Native application.
- Multi-currency groups.

## Architecture

Wallet will use two services:

1. **Frontend:** Next.js App Router PWA with Mantine.
2. **Backend:** NestJS REST API backed by PostgreSQL.

The frontend does not connect directly to PostgreSQL. It calls the NestJS API for all product data. The API owns authorization, validation, balance rules, and database transactions. This keeps the future React Native path clean: mobile can use the same REST API instead of depending on Next.js server actions or web-only internals.

Source code should be organized as a Bun workspace so the frontend and backend share one lockfile, shared TypeScript configuration, and common scripts. Bun is the default tool for dependency installation, local development, tests, builds, and container start commands. Frontend Next.js scripts should use Bun's Next.js guidance, e.g. `bun --bun next dev`, `bun --bun next build`, and `bun --bun next start`. Backend NestJS scripts should run through Bun-compatible commands, with implementation choices constrained to dependencies that work under Bun.

Coolify runs separate services for:

- Next.js frontend.
- NestJS API.
- PostgreSQL.

The frontend should be self-hosted as a dynamic Next.js app, not a static export, because authenticated screens and PWA behavior need runtime configuration. The PWA must include a manifest, app icons, service worker behavior, and stale-data indicators for offline read-only screens.

## Product Model

### Users and Groups

A user can belong to many groups. A group has exactly one owner and many memberships. Membership roles are:

- **Owner:** Manages group settings, ownership transfer, deletion, invites, envelopes, and all entries.
- **Admin:** Manages invites, envelopes, funding, transfers, reports, and group entries.
- **Member:** Views shared group data and records their own expenses. Member funding is allowed only when group settings permit it.

Invite links are generated by owners/admins. Links expire and can be revoked. A visitor accepting an invite must sign up or log in before joining the group.

### Envelopes

Wallet uses persistent envelopes, not monthly budgets. Each group has one currency. Envelopes hold running balances that carry forward indefinitely.

Envelope balance is derived from money movements:

```text
balance = funding + incoming transfers - outgoing transfers - expenses
```

Overspending is allowed by default. An overspent envelope shows a negative balance and appears in the dashboard attention area instead of blocking entry.

### Expenses and Recurring Expenses

Expenses are manually entered with:

- Group.
- Envelope.
- Amount.
- Date.
- Merchant/payee or short title.
- Optional note.
- Created-by member.

Recurring expenses represent expected future expenses such as rent or subscriptions. They appear in upcoming reminders and can generate expense entries when confirmed by a user. The MVP does not auto-charge, sync with banks, or settle debt between members.

## Primary UX

The dashboard is **envelope-first**. It answers “how much is left?” before showing analytics.

Primary navigation:

- Home.
- Envelopes.
- Activity.
- Reports.
- Settings.

Home shows:

- Total available across envelopes.
- Spending this month.
- Envelope cards with current balances.
- Overspent or low-balance envelopes.
- Upcoming recurring expenses.
- Recent activity.

Primary actions:

- Add expense.
- Fund envelope.
- Transfer between envelopes.
- Create recurring expense.
- Invite member.

The frontend displays API-provided balances and summaries. It may cache those responses for offline read-only mode, but it must clearly mark them as stale when offline or when the last refresh time is old.

## API and Data Flow

The frontend uses authenticated REST calls to the NestJS API. Mutations return the updated resource plus the recalculated envelope or dashboard summary needed to refresh the current screen.

Example API areas:

- `auth`: signup, login, logout, refresh session, current user.
- `groups`: create group, update settings, transfer ownership.
- `memberships`: list members, change role, remove member.
- `invites`: create, revoke, accept invite link.
- `envelopes`: create, update, archive, list balances.
- `funding`: add funds to envelope.
- `transfers`: move money between envelopes.
- `expenses`: create, update, delete, list expenses.
- `recurring-expenses`: create, update, list upcoming, confirm occurrence.
- `reports`: envelope spending, monthly spend, and envelope balance trends.

Every group-scoped request verifies membership before reading data. Every privileged mutation verifies role before writing data. Balance-changing operations run in database transactions.

## Data Storage

PostgreSQL is the source of truth. Core tables:

- `users`
- `groups`
- `memberships`
- `invites`
- `envelopes`
- `envelope_funding`
- `envelope_transfers`
- `expenses`
- `recurring_expenses`

Use database migrations for schema changes. Keep balance-affecting writes auditable: funding, transfers, and expenses should not silently disappear from history. Balance-affecting entries are soft-deleted or reversed instead of hard-deleted, with audit metadata recording who made the change.

## Authentication and Security

MVP auth is email/password. Passwords are hashed server-side. Sessions are issued by the NestJS API. The PWA uses an API-issued HTTP-only, Secure cookie for refresh/session continuity and a short-lived access token for API calls. The same auth endpoints should remain compatible with bearer-token storage in a future React Native client.

Security requirements:

- Validate all request bodies at the API boundary.
- Enforce group membership on every group-scoped read.
- Enforce owner/admin/member permissions on every mutation.
- Expire and revoke invite links.
- Do not expose whether an email address already has an account during public auth flows.
- Keep environment variables validated at API startup.

## Error Handling

User-facing errors should distinguish:

- Invalid form input.
- Login failure.
- Unauthorized or insufficient role.
- Expired or revoked invite link.
- Missing group or envelope.
- Stale offline cached data.
- Failed deployment dependency, such as API unavailable.

Overspending is not an error. It is a warning state on the affected envelope.

## Testing Strategy

Backend tests cover:

- Signup, login, and protected routes.
- Invite creation, revocation, expiry, and acceptance.
- Role permissions for owner, admin, and member.
- Group data isolation.
- Envelope balance calculations.
- Funding, transfers, expenses, and reversals/soft-deletes.
- Recurring expense occurrence confirmation.

Frontend tests cover:

- Signup/login flow.
- Create group flow.
- Accept invite flow.
- Create envelope.
- Fund envelope.
- Add expense.
- Transfer between envelopes.
- Offline read-only dashboard with stale-data banner.

Deployment smoke tests verify:

- Frontend reaches the API using configured base URL.
- API reaches PostgreSQL using configured database URL.
- API health endpoint reports readiness.
- Database migrations run before serving traffic.
- Bun scripts run frontend tests, backend tests, and build commands consistently in local and Coolify environments.

## Deployment

Coolify deployment uses:

- One Next.js frontend service.
- One NestJS API service.
- One PostgreSQL service.

Required configuration:

- Frontend API base URL.
- API database URL.
- API auth/session secrets.
- Allowed frontend origin.
- Cookie/security settings per domain.
- PostgreSQL backup policy.

The API exposes a health endpoint for Coolify checks. The database migration command runs during deployment before the API accepts traffic.

Both application services should install dependencies with `bun install --frozen-lockfile`. Dockerfiles or Coolify build commands should use Bun for builds and start commands rather than npm, pnpm, or yarn.

## Future React Native Path

The MVP should avoid web-only backend coupling. Product behavior belongs in the NestJS API, not in Next.js-only server actions. The future mobile app should reuse:

- Auth endpoints.
- Group/invite endpoints.
- Envelope, funding, transfer, expense, recurring expense, and report endpoints.
- Backend validation and permission rules.

The PWA can still use Mantine and web-specific PWA caching. Those choices must not leak into API contracts.
